## 1. 目标与范围

本阶段目标：在阶段 1 范围内交付一个能在 MT5 上稳定运行的 XAUUSD 剥头皮 EA，保留原阶段 1 的模块化分层架构和“三策略并行”主骨架，同时吸收 Hybrid Scalper 设计中最关键的实盘约束，使执行质量和持仓管理达到可用于实盘评估的水平。

本阶段包含以下能力：

1. 三个并行运行的独立子策略（EMA、Bollinger、RSI），保留各自信号内核与魔术号隔离
2. 市场状态识别与异常检测，作为环境上下文而非主信号源
3. 新增会话过滤，仅在伦敦盘与纽约盘开仓
4. 新增统一执行门控（ExecutionGuard），把点差、滑点、stop-level / freeze-level、冷却时间、日损和连亏纳入统一拦截
5. 新增 M5 高周期趋势确认，作为信号放行器，不替代原策略信号
6. 全策略统一的部分止盈 + 保本 + ATR trailing + 超时退出框架
7. 有限加仓，只对盈利中的同向仓位允许，带次数与总风险上限
8. 基于凯利公式的动态仓位管理，外加 0.5% 基础风险锚点与总风险上限
9. Tick 数据收集、指标管理、数据持久化、决策快照与完整日志
10. 绩效统计、执行质量统计、回测框架、HTML 报告与实时仪表盘

本阶段不包含：

1. 复杂的机器学习信号模型
2. 多品种联动
3. 外部经济日历 API 接入
4. 云端同步或远程控制
5. 将 EA 改写为完全以市场状态驱动的 Hybrid 引擎

阶段 1 保持阶段 1 原设计的分层架构：策略层、执行层、风控层、市场分析层、数据层、分析层、界面层。Hybrid 设计的能力以外层增强的形式吸收，不重写原策略内核。

### 2.2 运行数据流

信号放行顺序固定为：会话过滤 → ExecutionGuard → 策略原生信号 → M5 趋势确认 → 风控与仓位 → 执行 → 持仓管理 → 统计与日志 → UI。

```text
OnTick / OnNewBar
  -> CTickCollector
  -> CIndicatorManager
  -> CMarketAnalyzer              (输出市场状态，供 Guard 引用)
  -> CSessionFilter               (新增：先决定当前是否允许开仓)
  -> CExecutionGuard              (新增：统一拦截 spread/slippage/stops/freeze/cooldown/日损/连亏/异常市场)
  -> CStrategyEMA / CStrategyBollinger / CStrategyRSI  (产生原生候选信号)
  -> CTrendConfirm (M5)           (新增：按策略规则放行/拒绝)
  -> CRiskManager                 (基础风险锚点 + 凯利 + 仓位上限)
  -> CExecutionEngine             (市价/限价选择 + 重试 + 部分成交处理)
  -> CPositionManager             (统一出场 + 有限加仓)
  -> CPerformanceTracker
  -> CLogger
  -> CDashboard
```

### 2.3 核心设计原则

1. 三策略并行骨架不变，每个策略独立信号、独立统计、独立魔术号、独立持仓上限
2. Hybrid 设计的能力以外层增强方式接入，不改写策略内核
3. 风控前置：信号放行前先经过会话过滤与 ExecutionGuard
4. M5 趋势确认只作为“方向放行器”，不替代原策略信号
5. 市场状态是环境上下文，负责辅助放行、驱动下单方式、触发风险保护，不作为唯一交易入口
6. 关键决策点、执行质量、异常事件均持久化，可用于回测与实盘审计
7. 为后续阶段预留扩展点，但不在阶段 1 引入无实际需求的抽象

## 3. 策略层设计

### 3.1 策略组合方式

保留阶段 1 原设计：三个子策略并行，独立判断信号、独立开平仓、独立统计，不互相阻塞。

### 3.2 统一策略基类

`CStrategyBase` 保留阶段 1 原接口，至少提供：

- `CheckSignal()`
- `GetStopLoss()`
- `GetTakeProfit()`
- `ShouldClosePosition()`
- `UpdateStatistics()`
- `CalculateKellyFraction()`

公共状态包括：策略名称、魔术号、最大持仓数、是否启用、总交易数、盈利交易数、总盈利、总亏损。

### 3.3 策略 1：快速 EMA 交叉

保留阶段 1 原信号与止盈止损规则：

- EMA(5) 与 EMA(10) 交叉产生候选信号
- EMA(20) 作为方向过滤器
- 止损：ATR(14) × 1.5，止盈：ATR(14) × 1.2
- 最小止损 0.5 美元，最大止损 2.0 美元

### 3.4 策略 2：布林带突破回踩

保留阶段 1 原信号与止盈止损规则：

- 价格突破布林带上下轨后回踩入场
- 回踩距离 < 0.2 美元
- 突破伴随 Tick 量放大
- 固定止损 1.0 美元，固定止盈 0.8 美元

### 3.5 策略 3：RSI 极值反转

保留阶段 1 原信号与止盈止损规则：

- RSI(14) < 25 且回升做多；RSI(14) > 75 且回落做空
- 价格需位于 EMA(50) 附近，距离 < 1.5 美元
- 固定止损 0.6 美元，固定止盈 0.5 美元

### 3.6 策略隔离规则

1. 每个策略最多同时持有 1 单初始开仓，加仓由有限加仓模块另行管理
2. 各策略独立维护交易统计
3. 各策略使用独立魔术号
4. 允许不同策略同时持有相反方向头寸

### 3.7 M5 趋势确认与策略的协作规则

新增模块 `CTrendConfirm`，输出三种状态：

- `TREND_BULLISH`：M5 EMA20 > EMA50，EMA20 最近 3 根 M5 K 线斜率为正，收盘位于 EMA20 上方
- `TREND_BEARISH`：M5 EMA20 < EMA50，EMA20 最近 3 根 M5 K 线斜率为负，收盘位于 EMA20 下方
- `TREND_NEUTRAL`：其余情况

对三个策略的放行规则：

- **EMA 交叉**
  - 做多：仅在 `TREND_BULLISH` 或 `TREND_NEUTRAL` 放行；`TREND_BEARISH` 直接拒绝
  - 做空：仅在 `TREND_BEARISH` 或 `TREND_NEUTRAL` 放行；`TREND_BULLISH` 直接拒绝
- **布林带突破回踩**
  - 做多：仅在 `TREND_BULLISH` 放行；`TREND_NEUTRAL` 与 `TREND_BEARISH` 拒绝
  - 做空：仅在 `TREND_BEARISH` 放行；`TREND_NEUTRAL` 与 `TREND_BULLISH` 拒绝
- **RSI 极值反转**
  - 做多：`TREND_NEUTRAL` 直接放行；`TREND_BULLISH` 下仅在价格未明显远离 M5 EMA20 时放行；`TREND_BEARISH` 拒绝
  - 做空：镜像规则

M5 趋势确认只作用于“放行或拒绝”，不修改策略自身的止损止盈逻辑。

## 4. 市场分析层设计

### 4.1 市场状态

`CMarketAnalyzer` 输出四类市场状态：

- `MARKET_RANGING`
- `MARKET_TRENDING`
- `MARKET_BREAKOUT`
- `MARKET_ABNORMAL`

判断输入：ATR、ADX、布林带位置与带宽、近 20 根 K 线高低点突破次数、价格波动方差、Tick 数量与跳动质量。判断规则沿用阶段 1 原设计。

### 4.2 市场状态与策略的协作

阶段 1 中，市场状态是环境上下文，职责收敛为：

- `MARKET_ABNORMAL`：禁止所有策略新开仓，既有仓位仍由持仓管理退出
- `MARKET_BREAKOUT` / `MARKET_TRENDING`：EMA 与 Bollinger 正常参与；RSI 默认降权，仅在趋势强度未过阈值时允许
- `MARKET_RANGING`：RSI 优先；Bollinger 需满足突破回踩条件；EMA 继续运行，但信号优先级记为普通

### 4.3 异常检测与暂停策略

沿用阶段 1 原异常检测规则：

- ATR 暴涨：当前 ATR > 50 周期均值 × 2.5 时暂停交易 5 分钟，ATR 回落至均值 × 1.8 以下恢复
- 价差异常：价差 > 0.15 美元立即暂停，价差 < 0.08 美元持续 30 秒恢复
- 价格跳空：相邻 Tick 价差 > 0.5 美元暂停 3 分钟，连续 20 个 Tick 无异常恢复
- 流动性枯竭：连续 10 秒 Tick 数/秒 < 3，进入仅限价单模式
- 连续拒单：1 分钟内拒单率 > 50% 暂停 10 分钟并写告警

所有异常事件都必须写日志、在图表标注异常区域、纳入报告。

## 5. 执行门控与下单方式

### 5.1 会话过滤

新增 `CSessionFilter`，阶段 1 默认只允许在伦敦盘和纽约盘开新仓；非允许时段内既有仓位继续由持仓管理接管直到退出。具体服务器时间窗口通过外部参数配置，默认：

- 伦敦盘：按经纪商服务器时间的伦敦主时段
- 纽约盘：按经纪商服务器时间的纽约主时段
- 伦敦纽约重叠时段：正常交易

### 5.2 执行门控

新增 `CExecutionGuard`，在策略生成原生信号之前统一做“能不能交易”的前置拦截。它只消费 `CSessionFilter` 和 `CMarketAnalyzer` 提供的状态，不重复定义会话与异常规则。

检查项：

1. 点差是否超过阈值
2. 最近成交滑点与预估滑点是否过高
3. `SYMBOL_TRADE_STOPS_LEVEL` 与 `SYMBOL_TRADE_FREEZE_LEVEL` 是否满足
4. 是否处于策略冷却时间（由连续拒单、连续失败、连亏触发）
5. 是否达到单策略连亏暂停条件
6. 是否达到当日亏损阈值
7. 是否处于 `MarketAnalyzer` 标记的异常市场状态（引用，不重复判定）
8. 是否处于 `SessionFilter` 标记的非交易窗口（引用，不重复判定）

默认参数取向：按 ECN/Raw 账户优化，全部参数可调以兼容普通标准账户。所有 Guard 拒绝必须写 `guard_YYYYMMDD.log` 并带拒绝原因代码。

### 5.3 下单方式

保留阶段 1 原“市场状态驱动”的下单选择：

- `MARKET_TRENDING` / `MARKET_BREAKOUT`：优先市价单
- `MARKET_RANGING`：优先限价单
- `MARKET_ABNORMAL`：禁止新开仓

限价单（以下数值为默认参数，均可外部配置）：

- 做多限价单在当前 Ask 下方 0.1 美元挂出
- 做空限价单在当前 Bid 上方 0.1 美元挂出
- 挂出后默认 5 秒未成交撤销
- 撤销后必须重新评估市场条件，不允许机械重挂

### 5.4 执行保护机制

保留阶段 1 原规则：

- 滑点控制：偏离阈值标记为低质量执行，超过最大允许滑点取消重试
- 重试机制：单次执行失败最多重试 3 次，每次间隔约 100 ms
- 短时间内连续失败过多：对应策略暂停 1 分钟
- 部分成交：限价单部分成交后取消剩余未成交部分，避免碎片化仓位

## 6. 风控层与仓位管理

### 6.1 风控总原则

阶段 1 采用“策略独立仓位管理 + 全局风险上限 + 基础风险锚点”的三层约束。

### 6.2 基础风险锚点

- 单次初始开仓基础风险：账户净值的 **0.5%**
- 最大总开放风险：账户净值的 **5%**
- 达到总开放风险上限时禁止任何新开仓和加仓
- 凯利计算结果作为基础风险的调整因子，不能突破基础风险锚点对应的上限

### 6.3 凯利公式仓位模型

保留阶段 1 原结构：

- 冷启动：策略历史交易 < 30 单时使用预设胜率与盈亏比
  - EMA：胜率 55%，盈亏比 1.2
  - 布林带：胜率 48%，盈亏比 1.5
  - RSI：胜率 60%，盈亏比 0.9
- 真实统计：>= 30 单后使用真实统计，每 10 单更新一次凯利比例
- 使用半凯利 `f_actual = f × 0.5`
- 仓位 `Lot = 账户净值 × f_actual / 单笔止损金额`
- 最小手数 0.01，最大手数由参数限定
- 凯利值为负时禁止开仓

### 6.4 仓位控制规则

**策略级**

1. 单策略最多 1 单初始仓位
2. 同一策略已有初始仓位时，禁止再开新初始仓位
3. 同一策略在持仓存活期内允许有限加仓（详见 6.6）

**全局级**

1. 同时存在的初始仓位不超过 3 单（与阶段 1 原设计保持一致）
2. 含加仓子单在内的总持仓数不超过 9 单（3 策略 × 每策略 1 初始 + 2 加仓）
3. 所有持仓理论最大亏损之和不超过账户净值的 5%
4. 当总开放风险 + 新计划开仓的理论最大亏损超过 5% 上限时，新开仓或加仓被拒绝
5. 达到单日最大亏损阈值后禁止新开仓与加仓

**对冲规则**

- 允许多空对冲
- 当存在方向对冲时，在 UI 中给出明确提示

### 6.5 统一出场模型

三个策略统一使用以下分层出场：

1. **初始止损**：按各策略原规则计算
2. **部分止盈**：当浮盈达到 **1.0R** 时，先平掉 50%
3. **保本 + 缓冲**：部分止盈后，剩余仓位止损移动到保本价加小缓冲
4. **ATR trailing**：剩余仓位启用基于 ATR 的移动止损
5. **超时退出**：超过最大持仓时长仍未达到退出目标时，执行超时平仓

缓冲点数、ATR trailing 倍数、最大持仓时长均参数化。

### 6.6 有限加仓

只允许对盈利中的同向仓位加仓：

1. 最多额外加仓 **2 次**
2. 首单浮盈达到 **+0.5R** 才允许第 1 次加仓
3. 每次加仓都必须出现新的有效同向原生信号
4. 新加仓与上一次同向入场之间需满足最小价格距离
5. 加仓后该策略“交易战役”总开放风险不得超过其基础风险锚点的 **1.5 倍**
6. 下列任一条件成立时禁止继续加仓：
   - 首单尚未完成保本保护
   - 当前市场状态为 `MARKET_ABNORMAL`
   - M5 趋势确认不再支持该方向
   - 当日亏损或连亏阈值已触发
7. 加仓子单受全局风险上限约束

## 7. 数据层设计

### 7.1 Tick 数据收集器

沿用阶段 1 结构，每 Tick 记录至少：时间、Bid、Ask、Last、Volume、Spread、Flags。缓冲区默认容量 10000 条。

### 7.2 指标管理器

统一持有并更新以下指标句柄与缓冲区：EMA(5)、EMA(10)、EMA(20)、EMA(50)、Bollinger、RSI(14)、ATR(14)、ADX(14)、+DI / -DI，以及 M5 EMA20 与 EMA50（用于 M5 趋势确认）。

衍生指标：布林带宽度、价格在布林带中的位置、ATR 变化率、ADX 是否抬升、M5 EMA 斜率。

### 7.3 持久化结构

文件位置：`MQL5/Files/XAUUSD_Scalper/`

文件：

1. `trade_history.csv`
2. `tick_data_YYYYMMDD.bin`（回测模式下可选启用全量）
3. `statistics.json`
4. `decision_snapshots.csv`
5. `execution_quality.csv`（新增，汇总执行质量）

### 7.4 决策快照最小字段

每次候选信号评估时写入一条决策快照，至少包含：

- 时间
- 策略名
- 方向
- 候选信号是否成立
- 会话过滤是否通过
- ExecutionGuard 是否通过
- 市场状态
- M5 趋势状态
- 被拒绝原因代码
- 点差
- ATR
- ADX
- 计划止损距离
- 计划手数
- 是否为加仓判断

### 7.5 持久化策略分档

**实盘模式**

- 平仓后立即写交易历史
- 每 10 笔交易写一次统计数据
- 决策快照在信号触发、被拒绝、异常时写入
- Tick 数据只记录异常事件与统计摘要，不做全量落盘

**回测模式**

- 全量写交易历史
- 每笔交易后更新统计
- 决策快照记录所有策略评估时刻
- 可选启用全量 Tick 导出

**清理策略**

## 10. 关键运行流程

### 10.1 OnTick 主流程

1. 收集 Tick 数据
2. 更新指标缓存（含 M5 趋势确认指标）
3. 更新市场状态与流动性评分
4. 会话过滤检查
5. ExecutionGuard 检查（引用市场状态与会话状态）
6. 三个策略独立生成原生候选信号
7. M5 趋势确认放行
8. 风控计算仓位与基础风险锚点检查
9. 执行引擎处理订单
10. 更新持仓管理（部分止盈、保本、trailing、超时）
11. 处理有限加仓决策
12. 更新绩效统计（含信号质量、执行质量、持仓管理、加仓行为）
13. 写日志与决策快照
14. 更新 UI

### 10.2 开仓流程

1. 会话过滤通过（`CSessionFilter`）
2. ExecutionGuard 通过（含对异常市场状态与非交易时段的引用）
3. 策略产生原生候选信号
4. M5 趋势确认通过
5. 计算止损止盈与计划手数（基础风险锚点 + 凯利调整）
6. 通过策略级与全局级仓位/风险上限检查
7. 选择下单方式（市价或限价）
8. 执行订单并记录执行质量
9. 登记到持仓管理模块（统一出场与有限加仓）
10. 写日志与决策快照

### 10.3 平仓流程

平仓来源：

1. 止损止盈命中
2. 策略主动平仓条件触发
3. 统一出场模型的部分止盈、保本、trailing、超时
4. 异常市场触发的防御性退出
5. 持仓管理器判定应平仓

平仓后必须更新：策略统计、凯利输入数据、日志、交易历史、决策快照、图表标注、报告数据源。

### 10.4 有限加仓流程

1. 仅当存在该策略同向浮盈仓位且浮盈 >= 0.5R 时进入判断
2. 必须出现新的同向原生信号
3. 通过会话过滤、ExecutionGuard、M5 趋势确认
4. 通过基础风险锚点与全局风险上限检查
5. 满足与上次同向入场的最小价格距离
6. 达到加仓次数上限或首单未完成保本保护时，禁止继续加仓
7. 加仓子单纳入统一出场管理

## 11. 参数设计原则

所有关键参数作为外部输入暴露，但总数量需受控，保持可管理。暴露的主要参数类别：

1. 策略启用开关
2. EMA、RSI、布林带等指标参数
3. 固定与动态止盈止损参数
4. 部分止盈比例、保本缓冲、ATR trailing 倍数、最大持仓时长
5. 会话过滤时间窗口
6. ExecutionGuard：最大点差、最大滑点、日损阈值、连亏阈值、冷却时间
7. 最大重试次数与挂单超时
8. 凯利乘数、最大手数、最小手数、基础风险百分比、总风险上限
9. 有限加仓：最大次数、最小浮盈条件、最小价格距离
10. M5 趋势确认参数：EMA 周期、斜率窗口
11. 异常检测阈值
12. UI 开关与日志级别
13. 回测增强模式开关

### 12.3 阶段 1 验收标准

阶段 1 完成的最低验收标准：

1. EA 能在 MT5 正常编译运行
2. 三个策略可独立启停、独立统计、独立开仓
3. 仅在伦敦盘和纽约盘允许新开仓
4. ExecutionGuard 能稳定拦截高点差、stop-level/freeze-level 不满足、日损超限、连亏超限
5. M5 趋势确认影响信号放行，且每一次拒绝都可追踪到原因
6. 三个策略都使用统一的部分止盈 + 保本 + ATR trailing + 超时退出
7. 有限加仓只会发生在盈利同向仓位，总风险封顶有效
8. 基础风险锚点 0.5% 生效，凯利计算不会突破锚点对应的上限
9. 异常行情能触发自动暂停，并在恢复后自动恢复
10. 数据文件可正确写入与读取
11. 回测输出能区分“信号问题”“门控问题”“执行问题”“持仓管理问题”
12. 图表面板与交易标注正常显示

## 14. 风险与约束

### 14.1 业务风险

1. XAUUSD 在极端行情下的点差与滑点可能远超历史均值
2. 高频剥头皮对执行质量极度敏感
3. 回测结果可能因流动性建模不充分而高估实盘表现
4. 有限加仓在单边趋势结束时可能放大回撤

### 14.2 工程约束

1. MQL5 对复杂抽象与外部依赖支持有限
2. Tick 级全量记录可能产生较大文件体积
3. 图表对象过多可能影响界面性能
4. 策略、Guard、M5 确认之间的交互会显著提高测试复杂度

### 14.3 设计应对

1. 风控前置：SessionFilter 与 ExecutionGuard 在策略执行之前
2. 异常检测优先级最高
3. 回测框架重点覆盖执行质量与市场状态
4. 数据记录分层，避免实盘无节制写盘
5. UI 与分析尽量增量更新，减少全量重绘
6. 加仓受“基础风险锚点 × 1.5”硬约束

## 15. 最终结论

阶段 1 合并版在保留阶段 1 原蓝图“模块化分层 + 三策略并行 + 凯利仓位 + 市场状态识别 + 完整分析框架”的基础上，吸收 Hybrid Scalper 设计中最具实盘价值的 4 类能力：会话过滤与执行门控、M5 高周期趋势确认、统一出场管理、有限加仓。

这不是一次重写，而是一次“结构不动、实盘约束补齐”的升级。完成后，阶段 1 既能作为剥头皮 EA 的稳定底座，也能为后续阶段引入更复杂的 regime 引擎、多时间框架信号与更精细的执行与风险模型提供清晰可扩展的接口。
